## 接口鉴权

### 鉴权接口

#### 接口说明

审计业务场景对外提供的接口均需要鉴权，鉴权的认证获取 accessToken 是调用业务接口的第一步，相当于创建了一个登录凭证。所有三方业务API接口，都需要依赖于 accessToken 来鉴权调用者身份。

本接口用于获取 accessToken 鉴权凭证。通过 appKey 和 secret 获取 accessToken 凭证。appKey 和 secret 由项目交付人员提供。

accessToken 具有时效性，第一次调用此接口会产生一个30天有效期的 accessToken，再次调用仍会返回相同的 accessToken，如果在距离 accessToken 过期时间小于七天时，调用此接口会重新生成新的 token。

关于调用方对 accessToken 的开发要求：

A. 固定每天触发获取 token 接口，并保存到自有系统。如果调用方有多个子系统在同时调用，需要注意 accessToken 在多个系统之间保持一致。

B. 当任意接口收到 token 过期失效反馈时，触发获取 token 接口重新获取。

当任意接口鉴权失败时，会收到如下报文：

```json
{
    "traceId": "062cd9dae3da4076833e8a2035e2dd9a",
    "code": 602,
    "message": "appKey或者accessToken缺失/无效"
}
```

注意：为了安全考虑，****开发者 请勿 在web端使用鉴权访问接口****，相关鉴权密钥需要开发者保存在后台。

#### 接口请求

**请求方式**：POST（HTTPS）

**请求地址**：https://domain:port/mmba/api/user/**v3**/accessToken

#### 请求头参数

| 参数 | 类型 | 是否必填 | 描述 |
| --- | --- | --- | --- |
| Api-Info | string | 是 | 租户code（明文传输） |
| Content-Type | string | 是 | application/json |

#### 请求体参数说明

| 参数 | 类型 | 是否必填 | 描述 |
| --- | --- | --- | --- |
| companyCode | string | 是 | 公司标识（租户名） |
| appKey | string | 是 | 分配的appKey |
| secret | string | 是 | 分配的secret |

#### 返回参数说明

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| code | int | 200,601,602 (参看错误码对照表) |
| accessToken | string | 身份验证凭证，有效期30天 |

#### 请求示例

```bash
curl -k -XPOST 'https://domain:port/mmba/api/user/v3/accessToken' \
-H 'Content-Type: application/json' -H'Api-Info: mmba' \
-d '{"companyCode":"mmba","appKey":"1_8e0*****426a4","secret":"395*****5ef2d"}'
```

## 调用流程

### 指令发送

> 💡 指令发送是API接口的一项重要能力，能够通过接口层面对手机端下发指令，实现相关动作如拨打电话、短信，微信发送消息等能力。

指令发送的数据流主要包括了**三方业务系统、API系统、审计系统、客户端**等多个系统的数据交互。

当三方发起业务请求时，由API系统进行接口校验，通过后经由审计系统向客户端发送指令，客户端收到指令后执行指令。

客户端会将指令执行结果上报，经由API系统回调给三方系统。

### 数据回调

> white_check_mark **使用介绍：**三方系统需要提供一个http服务，用于接收审计系统发送的回调数据。

**回调地址配置：**

- **回调地址由客户提供**，****由我司交付人员协助在管理后台进行配置。****
- 回调服务地址配置完成后，当客户端生成数据并上报到审计平台时，用户配置的地址会收到具体数据。
- 可以配置多个地址，每个地址收到的数据是一致的，地址越多效率越差，建议不多于2个。

**数据回调包含2类大的业务：**

- 指令回调，表示指令执行结果的数据回调；
- 审计数据回调，表示由客户端审计到的数据最终回调给三方系统的流程

**数据回调规则说明：**

> - 未配置回调地址时，无数据；配置回调地址时，有数据
- 数据由配置回调地址那一刻之后开始产生，不包含历史数据
- 回调地址对应的接口在接到数据后，如果返回http状态码为200且数据体不为空，则视为数据接收成功，否则1分钟后才会推送新数据。所有数据以时间先后顺序排队回调，****如果接口一直不通，则会阻塞后面的数据回调，并在本系统产生数据积压。********接收方应考虑避免自身系统数据处理逻辑时间过长导致接口超时，进而阻塞数据回传。****
- 回调接口的连接超时时间为10秒，数据读取超时时间为20秒，建议异步处理回调数据。
- 接收成功的数据，会从推送通道中删除，不做留存
- 回调的data是一个列表，一次最多回调20条数据
- 同一条记录有可能会回传多次，接收方应自行对数据重复性做校验，一般以esId字段唯一标识记录。

### SM4对称加密规则

- 密钥（参数 16个字符 英文大小写字母及数字）
- 初始化向量IV (参数 16个字符 英文大小写字母及数字)
- 加密参数MD5，密钥拼接初始化向量后的字符串做MD5计算
- 加解密模式：SM4/CBC/PKCS5Padding

### AES对称加密规则

- 密钥（参数 16个字符 英文大小写字母及数字）
- 初始化向量IV（参数 16个字符 英文大小写字母及数字）
- 加解密模式：AES/CBC/PKCS5Padding
- BASE64加解密：加密后的字节流在传输前做BASE64转码

## 接口协议

### 协议介绍

所有的接口需使用HTTP(S)协议传输、POST方法。

- **数据体**：默认**数据体采用JSON数据格式。**
- **数据头**：需要填写 Api-Info 头信息，是必填项。
- 加密报文情况下，Content-Type为text/plain，非加密情况下为application/json。

### 请求样例

**请求头如下：**

```json
Api-Info:mmba
Content-Type:application/json
```

**curl指令示例如下：**

```bash
-H 'Content-Type: application/json' -H 'Api-Info: mmba'
```

### 返回样例

返回体采用Json数据格式

```json
{
    "traceId": "951a408f0aac4d9e9e86a55e68705a45",
    "code": 200,
    "message": "接口请求成功"
}
```

### 指令发送协议

指令发送协议由校验头、URL接口地址、数据体组成：

```bash
curl --location 'https://domain:port/mmba/api/phone/v3/dial' \
--header 'Content-Type: application/json' --header 'Api-Info: mmba'  \
--data '{...}'
```

### 数据回调协议

回调协议由请求头、三方系统URL接口地址、数据体组成：

```json
{
    "behaviorType": 13, 
    "data": [...],
    "tenancyName": "test"
}
```

## 用户绑定

**请求地址**：https://domain:port/mmba/api/user/**v3**/opUserEx

> exclamation **注：请求时必须携带请求头**

### 请求参数说明

| 参数 | 类型 | 是否必填 | 长度 | 描述 |

| accessToken | string | 是 | 255 | 身份验证中获得的accessToken |
| appKey | string | 是 | 80 | 分配的appKey |
| companyCode | string | 是 | 30 | 公司标识（租户名） |

### 请求参数说明

| 参数 | 类型 | 长度 | 是否必填 | 描述 |
